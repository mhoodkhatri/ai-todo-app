# Evolution of Todo — Hackathon II Constitution

## Core Principles

### I. Spec-Driven Development (NON-NEGOTIABLE)

All implementation MUST originate from written specifications. No manual
coding is permitted — code MUST be generated by Claude Code from refined
specs using the Spec-Kit Plus workflow.

- Every feature MUST have a Markdown specification before implementation
  begins.
- The workflow MUST follow: Write Spec → Generate Plan → Break into
  Tasks → Implement via Claude Code.
- If Claude Code generates incorrect output, the spec MUST be refined
  until the output is correct. Direct code edits are forbidden.
- All specification files MUST reside under `specs/<feature>/` with
  `spec.md`, `plan.md`, and `tasks.md` as the primary artifacts.
- Prompt History Records (PHRs) MUST be created for every user prompt
  to maintain a complete audit trail of the development process.

### II. AI-First Implementation

AI tools are the primary instruments for code generation, operations,
and deployment. Engineers act as Product Architects, not syntax writers.

- Claude Code with Spec-Kit Plus MUST be the primary code generation
  tool across all phases.
- OpenAI Agents SDK and Official MCP SDK MUST be used for the AI
  chatbot implementation (Phase III onward).
- AIOps tools (kubectl-ai, kagent, Docker Gordon) SHOULD be used for
  Kubernetes and Docker operations where available.
- All AI-generated code MUST be reviewed against the spec before
  acceptance.

### III. Phased Incremental Evolution

The project MUST evolve through five defined phases, each building on
the previous. No phase may be skipped.

- **Phase I**: In-Memory Python Console App (Basic Level features).
- **Phase II**: Full-Stack Web App (Next.js + FastAPI + Neon DB +
  Better Auth).
- **Phase III**: AI-Powered Todo Chatbot (OpenAI ChatKit + Agents SDK +
  MCP Server).
- **Phase IV**: Local Kubernetes Deployment (Docker + Minikube + Helm).
- **Phase V**: Advanced Cloud Deployment (Kafka + Dapr + AKS/GKE/OKE).
- Each phase MUST produce a working, demonstrable deliverable.
- The following bonus features MUST be implemented:
  - **Reusable Intelligence**: Create and use reusable intelligence via
    Claude Code Subagents and Agent Skills.
  - **Cloud-Native Blueprints**: Create and use Cloud-Native deployment
    Blueprints via Agent Skills.
  - **Multi-language Support**: The AI chatbot MUST support Urdu in
    addition to English.
  - **Voice Commands**: Voice input MUST be supported for todo
    commands.
- Features MUST progress through three levels across the phases:
  - **Basic Level** (Phase I–III): Add Task, Delete Task, Update Task,
    View Task List, Mark as Complete.
  - **Intermediate Level** (Phase V): Priorities & Tags/Categories,
    Search & Filter, Sort Tasks.
  - **Advanced Level** (Phase V): Recurring Tasks, Due Dates & Time
    Reminders.

### IV. Cloud-Native Architecture

From Phase II onward, the system MUST be designed for cloud-native
deployment with stateless services and event-driven patterns.

- The chat API server MUST be stateless — all conversation state MUST
  be persisted to the database.
- MCP tools MUST be stateless and store all state in the database.
- Event-driven architecture via Kafka MUST be adopted in Phase V for
  decoupled services (notifications, recurring tasks, audit logging,
  real-time sync).
- Dapr MUST be used in Phase V to abstract infrastructure dependencies
  (Pub/Sub, State, Bindings, Secrets, Service Invocation).
- All services MUST be containerized with Docker and deployable via
  Helm charts on Kubernetes.

### V. Clean Code & Project Structure

All code MUST follow clean code principles with proper structure,
regardless of being AI-generated.

- Windows users MUST use WSL 2 (Ubuntu) as the development
  environment. All CLI tools, builds, and deployments MUST run inside
  WSL 2.
- Python code MUST target Python 3.13+ and use UV as the package
  manager.
- Frontend code MUST use TypeScript with Next.js App Router and
  Tailwind CSS.
- Backend code MUST use FastAPI with SQLModel ORM and Pydantic models
  for request/response validation.
- All API routes MUST follow RESTful conventions under `/api/`.
- Errors MUST be handled with appropriate HTTP status codes
  (HTTPException in FastAPI).
- Secrets and tokens MUST never be hardcoded — use `.env` files and
  environment variables.
- **JWT Security Model (Phase II onward)**: Better Auth on the Next.js
  frontend MUST issue JWT tokens via its JWT plugin. The FastAPI backend
  MUST verify every request using JWT middleware with a shared secret
  (`BETTER_AUTH_SECRET` env var, identical in both services). All API
  endpoints MUST require a valid `Authorization: Bearer <token>` header;
  requests without a token MUST receive `401 Unauthorized`. Every
  database query MUST filter by the authenticated user's ID to enforce
  user isolation — no user may see or modify another user's tasks.

### VI. Monorepo Organization

From Phase II onward, the project MUST use a monorepo structure to
give Claude Code full context over frontend and backend in a single
workspace. Phase I uses a simple Python project layout (`/src`,
`README.md`, `CLAUDE.md`) with no frontend or monorepo overhead.

- **Phase I structure**: `/src` folder with Python source code, root
  `CLAUDE.md`, `README.md`, constitution, and specs.
- **Phase II+ structure**: Root `CLAUDE.md` MUST provide project
  overview and spec navigation.
- `frontend/CLAUDE.md` MUST define frontend-specific patterns and
  conventions.
- `backend/CLAUDE.md` MUST define backend-specific patterns and
  conventions.
- Specs MUST be organized under `specs/` with subdirectories for
  features, API, database, and UI.
- The `.specify/memory/constitution.md` file (this document) is the
  authoritative governance document for the project.

## Development Workflow

Every feature implementation MUST follow this workflow without
exception:

1. **Specify**: Write or update the feature spec at
   `specs/<feature>/spec.md`.
2. **Plan**: Generate the implementation plan at
   `specs/<feature>/plan.md` using `/sp.plan`.
3. **Task**: Break the plan into testable tasks at
   `specs/<feature>/tasks.md` using `/sp.tasks`.
4. **Implement**: Execute tasks via Claude Code — no manual coding.
5. **Validate**: Verify the implementation against acceptance criteria
   in the spec.
6. **Record**: Create a PHR for every prompt exchange during the
   process.
7. **Submit**: Ensure GitHub repo, published app link (Vercel), and
   demo video (≤90 seconds) are ready for each phase deadline.

### Deliverables Per Phase

Each phase submission MUST include:
- Public GitHub repository with constitution, specs, history, source
  code, and README.
- Working application demonstrating all required features for the
  phase.
- Published app link (Vercel) where applicable.
- Demo video under 90 seconds.

## Governance

This constitution is the authoritative document governing all
development practices in this project. It supersedes any conflicting
guidance in other files.

- **Amendments** require: (1) documentation of the change rationale,
  (2) update to this file, (3) a Sync Impact Report (HTML comment at
  top of file), and (4) version increment per semantic versioning.
- **Versioning**: MAJOR for principle removals/redefinitions, MINOR for
  new principles or material expansions, PATCH for clarifications and
  wording fixes.
- **Compliance**: All PRs and code reviews MUST verify adherence to
  these principles. Non-compliance MUST be flagged and resolved before
  merge.
- **ADR requirement**: Architecturally significant decisions (framework
  choices, data model changes, deployment strategies) MUST be
  documented via ADR upon user consent.
- **Runtime guidance**: See `CLAUDE.md` at project root and subfolder
  levels for development guidance that supplements this constitution.
